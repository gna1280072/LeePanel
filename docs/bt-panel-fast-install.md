# 宝塔面板"极速安装"原理分析

## 编译安装 vs 极速安装

宝塔面板提供两种 PHP 安装方式：

### 编译安装（慢，5-15 分钟）

```
下载源码 → install_deps → ./configure → make → make install → PECL 扩展
         ~~~~1min~~~~   ~~~30s~~~  ~~~5-10min~~~  ~~30s~~   ~~~2-5min~~~
```

整个过程 **make 编译**是最耗时的，CPU 要处理几百万行 C 代码。

### 极速安装（快，1-2 分钟）

宝塔提前在各种系统上编译好 PHP，打包成 **tar.gz 二进制包**，放在自己的 CDN 上：

```
下载预编译包 → 解压到 /www/server/php/8.2/ → 注册 systemd → 完成
  ~~30-60s~~        ~~~~5s~~~~                ~~~1s~~
```

跳过了整个编译过程，直接用编译好的二进制文件。

## 预编译包结构

### 宝塔 CDN 上的预编译包（举例）

```
https://download.bt.cn/install/rpm/php-8.2.32-centos9-x86_64.tar.gz
https://download.bt.cn/install/deb/php-8.2.32-ubuntu22-amd64.tar.gz
```

### 包内目录结构

```
/www/server/php/8.2/
├── bin/php              ← 已编译好的 PHP CLI
├── sbin/php-fpm         ← 已编译好的 PHP-FPM
├── etc/php.ini          ← 默认配置
├── lib/php/extensions/  ← 所有 .so 扩展都已编译好
│   ├── redis.so
│   ├── imagick.so
│   ├── swoole.so
│   └── ...
└── ...
```

## 对比权衡

| 对比项 | 极速安装 | 编译安装 |
|--------|---------|---------|
| **速度** | 1-2 分钟 | 5-15 分钟 |
| **安装方式** | 下载预编译二进制包 | 从源码编译 |
| **扩展灵活性** | 只能用包里的扩展 | 可自选扩展 |
| **系统兼容性** | 需要每种系统+架构都有预编译包 | 任何系统都能编译 |
| **性能** | 通用编译参数（-O2） | 可针对当前 CPU 优化（-march=native） |
| **维护成本** | 宝塔要维护大量预编译包 | 无维护成本 |

## 预编译包矩阵

宝塔需要维护的预编译包数量：

```
PHP 版本 × 系统 × 架构 = 包数量

6 个 PHP 版本 (7.4/8.0/8.1/8.2/8.3/8.4)
× 5 个系统 (CentOS 7/8/9, Ubuntu 20/22)
× 2 个架构 (x86_64, arm64)
= 60 个预编译包
```

每次 PHP 发布新版本，宝塔都要编译 60 个包上传到 CDN，这是极速安装的维护成本。

## LeePanel 是否需要做极速安装？

### 短期不需要

1. **维护成本太高** — 需要 CI/CD 自动化编译流水线，每种系统+版本组合都要打包
2. **没有 CDN** — 预编译包要放在服务器上供下载，需要带宽
3. **编译安装已经够用** — 服务器面板不是面向终端用户的产品，管理员可以接受 5-15 分钟

### 长期可以考虑的方案

- 用 GitHub Actions 做 CI，在 Docker 容器中编译各版本的 PHP
- 编译好的包上传到 GitHub Releases 或对象存储（如阿里云 OSS）
- 安装时优先下载预编译包，下载失败再回退到源码编译

这是一个完整的分发体系，工作量不小，建议作为后续版本的功能。
