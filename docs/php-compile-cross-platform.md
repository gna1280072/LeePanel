# 源码编译 PHP 的跨系统兼容性

## 结论

**不能通用**，编译出来的 PHP 是绑定当前系统的。

## 为什么不能跨系统用

编译时链接的是当前系统的库文件，不同发行版的库版本不同：

| 差异点 | CentOS | Ubuntu |
|--------|--------|--------|
| libc 版本 | glibc 2.28 (CentOS 8) | glibc 2.35 (Ubuntu 22.04) |
| 库路径 | `/usr/lib64/` | `/usr/lib/x86_64-linux-gnu/` |
| OpenSSL 版本 | 可能 1.1.1 | 可能 3.0 |
| libxml2 版本 | 可能 2.9.7 | 可能 2.12 |

把 CentOS 编译的 PHP 拿到 Ubuntu 上跑，大概率报 `libxxx.so: cannot open shared object file`。

## 同一发行版内兼容性

| 场景 | 是否兼容 |
|------|----------|
| 同版本（都是 Ubuntu 22.04 x86_64） | 基本可以 |
| 跨版本（Ubuntu 20.04 → 22.04） | 不一定，glibc 升级可能导致不兼容 |
| 跨架构（x86_64 → ARM） | 完全不行 |

## 宝塔的跨系统做法

宝塔的"跨系统"是靠**在每个系统上分别编译**实现的，不是编译一次到处用：

- 检测到 CentOS → 下载 CentOS 的编译脚本 → 在当前机器上编译
- 检测到 Ubuntu → 下载 Ubuntu 的编译脚本 → 在当前机器上编译

所以宝塔安装 PHP 慢，就是因为**每次都在目标机器上现场编译**。

## 真正能跨系统的方案

只有**静态编译**（把所有库打包进二进制）才能跨系统，但 PHP 官方不推荐：

- 二进制体积巨大
- 安全漏洞无法通过更新系统库来修复
- 很多扩展不支持静态链接

## 总结

源码编译 ≠ 跨平台，它只是解决了"路径统一"的问题，没有解决"系统库依赖"的问题。要跨系统，要么每个系统分别编译，要么用 Docker。
