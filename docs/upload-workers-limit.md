# Upload Workers 并发上限分析

**结论：设太大不会卡死服务器。**

## 各层面风险评估

| 层面 | 风险 | 说明 |
|------|------|------|
| SSH 连接 | 无 | 所有 worker 共享**同一条** SSH 连接，多路复用 SFTP 通道，通道数再多也不影响 sshd 稳定性 |
| 服务器 CPU | 极低 | SFTP 是 I/O 密集操作，几乎不消耗 CPU |
| 服务器内存 | 极低 | 每个 SFTP 通道缓冲区仅几 KB |
| 磁盘 I/O | 低 | 并发写不同文件/目录，Linux 内核调度处理 |
| 带宽 | 唯一瓶颈 | 所有 worker 共享同一条连接的带宽，worker 太多只是**每个分到的带宽变少**，速度变慢但不会崩溃 |

## 实际效果

| Worker 数 | 效果 |
|-----------|------|
| 3-16 | 正常范围，充分管道利用 |
| 50-100 | 每个 worker 分到的带宽很少，总速度反而可能下降，但不会卡死 |
| 500+ | 几乎没意义，SSH 通道调度开销开始显现，但 sshd 仍然不会崩溃 |

## 总结

设太大只是**浪费**，不会**损坏**。真正的上限是你的 SSH 连接带宽。
